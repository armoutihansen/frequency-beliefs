## Major

### M1. `parse_report` is imported but never used in `design_efficiency.py`

`scripts/design_efficiency.py` line 62 imports `parse_report` from `utils`,
but the symbol is not used anywhere in the codebase (only the definition in
`utils.py` and the import in `design_efficiency.py` appear in grep output).
That makes the consolidation a leaky abstraction --- a future reader will
wonder which CSV-reading code path needs it. Either:

- remove the unused import (and possibly the function, since
  `consistency_check.py` reads CSVs with `csv.DictReader` but never parses the
  `report` field --- it's a display column), or
- add a TODO note in `utils.py` documenting that it is exported for downstream
  notebook/analysis use.

If `parse_report` is meant for analysts re-reading `fixed_reports.csv` /
`draw_level.csv`, document that in the docstring; right now the function is
orphaned.

### M2. `consistency_check.py` was *not* wired into the new `config.py` despite the user noting it as a possible gap --- confirming the gap

The user flagged `CONSISTENCY_CSV_TOL` and `CONSISTENCY_ROUND_TOL` as defined
but unused. Confirmed --- `consistency_check.py` still uses bare literals:

- `1e-9` at lines 52, 63 (corollary checks, math identities --- arguably
  belong to `FLOAT_EQUALITY_TOL` semantically rather than CSV tolerance);
- `1e-9` at lines 95, 98 (alpha equality across CSV rows --- this is the CSV
  parsing tolerance);
- `0.01` at lines 194, 258 (manuscript-rounded-number tolerance).

The `0.01` literal in particular is exactly what `CONSISTENCY_ROUND_TOL = 0.01`
documents. Wiring the constants in would (a) make the audit table at the top of
`consistency_check.py` legible, (b) let `tests/test_consistency.py` reproduce
the same tolerance assumption without re-deriving it.

Recommend either:
- import `FLOAT_EQUALITY_TOL`, `CONSISTENCY_CSV_TOL`, `CONSISTENCY_ROUND_TOL`
  into `consistency_check.py` and replace the literals; or
- delete the two unused `CONSISTENCY_*` constants from `config.py` until they
  are wired (otherwise they document a falsehood --- "used by
  `consistency_check.py`" --- which the comment block above them now suggests).

### M3. Exploration scripts still use literal seeds despite the `SEED_HAMMING` / `SEED_CANDIDATE_RULES` constants

`config.py` defines `SEED_HAMMING = 20260521` and
`SEED_CANDIDATE_RULES = 20260522`. Grep confirms:

- `scripts/exploration/hamming_interior_search.py:156` --- `random.Random(20260521)`
- `scripts/exploration/hamming_spike.py:405` --- `np.random.default_rng(20260521)`
- `scripts/exploration/candidate_rule_checks.py:22` --- `random.seed(20260522)`

So the user's audit on the seed values is correct, but the *whole point* of
the constants was to remove the literals. The exploration scripts are
stdlib-only (per the user's note) and currently can't `from config import`
without the `sys.path` shim (only `hamming_spike.py` has the shim). Two
options:

- Add the shim to the other two scripts and import from `config`, paying the
  price of breaking pure-stdlib status.
- Drop the constants from `config.py` and let the literals stand --- they are
  already documented in each script's preamble.

The second is cleaner for a research repository; "constant lives next to use"
is fine when the use is a single literal in a single file. The current state
(constants defined but not imported) is the worst of both worlds.

Note also `hamming_spike.py` has *two* `np.random.default_rng` calls: the
main run uses `20260521`, and `self_check_forward` uses an internal `11`
(line 245) which is fine but is not the seed `SEED_HAMMING` would describe.
If the constants stay, document that `SEED_HAMMING` is for the main run only.

### M4. Notebook table_robust does not match the paper's column order

The notebook's robustness cell does:

```python
.pivot(index="alpha_label", columns="rule", values="win_share")
```

`pandas.pivot` sorts column labels alphabetically, producing columns
`(Discrete metric, Manhattan distance, Quadratic distance)`. The paper's
appendix table `tab:regime-wins-asymmetric` orders them
`(Discrete, Quadratic, Manhattan)` --- the project's canonical D/Q/M order, as
established in `consistency_check.py:198-199`. A non-coding reader comparing
the notebook to the appendix will see the same numbers in the wrong column
positions and may believe the regime story is reversed (e.g. for
`one-dominant`, the notebook shows D 0.06 / M 0.33 / Q 0.60; the paper reads
D 0.06 / Q 0.60 / M 0.33).

Fix: append `.reindex(columns=["Discrete metric", "Quadratic distance", "Manhattan distance"])`
to the pivot. Same issue applies to `table1` and `table2` --- check whether
the notebook's column order matches the paper for those as well; the user's
"non-coding reader would get confused" risk is real here.

### m3. `pyproject.toml` Python pin

`requires-python = ">=3.14"` is unusually strict for a research artifact
(Python 3.14 was released October 2025; many reviewers' machines will still
be on 3.12 or 3.13). Unless the codebase relies on a 3.14-only feature
(union types use `|` which is 3.10+; `from __future__ import annotations` is
present), `>=3.11` would maximise reproducibility surface. Confirmed by grep
that no PEP 695 generics (3.12+ only) or PEP 750 t-strings (3.14) are used.

### m4. `tests/test_consistency.py` timeout

`timeout=120` seconds for the consistency subprocess is tight on a slow CI
machine because `check_functional_menu` uses `c_denom=4000` for several
threshold reproductions. If that timeout fires the test failure will be
misattributed to a math drift rather than a slow runner. Suggest
`timeout=300`. (The user reports the local suite at ~108s, so 120 has very
little headroom.)

### m5. `Makefile install` target

`uv sync` installs the runtime dependencies, but the `dev` dependency-group
(`jupyter`, `pandas`, `pytest`) is in a separate group and `uv sync` won't
install it unless `--all-extras` or `--group dev` is passed (depending on
your uv version's semantics). If `make test` fails on a fresh checkout with
"no module named pytest," `make install` is the wrong escape hatch. Either
`uv sync --group dev` or document explicitly.

### m6. `Makefile reproduce` runs `simulate` which overwrites outputs

`reproduce: install verify simulate consistency paper` will rerun
`design_efficiency.py --final --draws 5000`, overwriting
`outputs/design_exercise/{rule_comparison.csv,latent_aggregate.csv,fixed_reports.csv}`
and the summary at `_context/simulation_results_summary.md`. For a referee

### m7. `notebooks/reproduce_analysis.ipynb` `ROOT` resolution

```python
ROOT = Path.cwd().parent if Path.cwd().name == "notebooks" else Path.cwd()
```

This is fragile: it relies on the *name* of the parent directory being
`notebooks`. If a user renames the directory or launches Jupyter from a
clone with a different parent layout, `ROOT` will silently point to the
wrong place. A more robust pattern is to anchor on the notebook's own
location, e.g.

```python
NB_DIR = Path(__file__).resolve().parent if "__file__" in globals() else Path.cwd()
```

(but Jupyter doesn't expose `__file__`; an alternative is to look for a
sentinel file like `pyproject.toml` walking up the tree).

## Notes

### Reconciliation of the two `clean` implementations

The user asked specifically whether the two pre-refactor `clean`
implementations (one returning `float(v)`, the other returning bare `v`)
were correctly reconciled. The current `utils.clean`:

```python
def clean(v: float, tol: float = 1e-10) -> float:
    if abs(v) < tol: return 0.0
    if abs(v - 1.0) < tol: return 1.0
    return float(v)
```

returns `float(v)` in the fallthrough --- the safer of the two prior
implementations. Effect: where the pre-refactor `design_efficiency.py`
preserved `numpy.float64` (or similar) types, the consolidated version now
casts to Python `float`. This is functionally equivalent for all CSV
output, summary tables, and the `BoundResult` dataclass (which doesn't
care about float subtype). I see no behavioural drift, and the test suite
agrees.

### `fmt_interval` None-handling

Call sites:

- `design_efficiency.py:717,720,722` --- never pass `None`; the `BoundResult`
  contract guarantees coords/means are populated.
- `verify_regions.py:834-835` --- guards with `if coord and i < len(coord)
  else "NA"`. Pre-`fmt_interval` is called only when both checks pass, so
  `fmt_interval` itself never sees `None` from this site; the `mean` arg at
  `:835` *can* be `None` (e.g. `l1_grid_bounds` returns `(None, None, 0)`
  when empty), and `fmt_interval` correctly returns `"NA"`.
- `verify_regions.py:860` --- `[fmt_interval(iv) for iv in (ex['l1_coord_grid'] or [])]`
  iterates over a list; individual entries are tuples, not None. Mean is
  passed directly, can be None, handled.
- `verify_regions.py:1228-1229` --- guarded by `if coord else 'NA'`; never
  calls with None.

Conclusion: None-handling is consistent. The function handles None
correctly; no caller depends on a non-None argument.

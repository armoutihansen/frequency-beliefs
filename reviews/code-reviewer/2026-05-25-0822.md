## 2. Purpose of code

- `verify_regions.py`: finite-enumeration / grid checks for the analytical characterizations (discrete-metric, squared-distance, L1/Manhattan, Hamming binary), plus risk-aversion counterexamples. Emits a markdown verification table and worked examples.
- `design_efficiency.py`: latent-belief design exercise. Draws Dirichlet beliefs, computes each rule's optimal report, then computes identification bounds. Writes `latent_aggregate.csv`, `rule_comparison.csv`, `fixed_reports.csv`, plus an `--save-draws` option.
- `consistency_check.py`: cross-checks Corollary 1 analytically and reproduces the manuscript's tables (regime-wins, regime-regret, design-nk) from `outputs/design_exercise/`.
- `candidate_rule_checks.py`, `hamming_interior_search.py`, `hamming_spike.py`: exploration-mode screenings backing ADR-0001 (Hamming excluded from horse race).

## Summary (<400 words)

The computational pipeline is in good shape. `verify_regions.py` runs cleanly under `uv run` and 13/13 checks pass. `consistency_check.py` reproduces every numerical claim in `06_design_comparison.tex` and `tab:design-nk` to displayed precision from `outputs/design_exercise/rule_comparison.csv` and `latent_aggregate.csv`. The simulation seed (`20260508`) is fixed; tie rates are 0 across all cells; payment probability is correctly emitted only for the discrete-metric rule, so it is not part of the cross-rule horse race.

Two material inconsistencies stand out:

1. The README still says "The final CSV outputs are stored in `outputs/simulation_design/`" but the manuscript's tables (in particular the regret figures) can only be produced from `outputs/design_exercise/`: the legacy `simulation_design/rule_wins.csv` lacks the `mean_regret` column and uses the older grid-based Manhattan coordinate algorithm, while `design_exercise/` uses the audited sharp threshold root-find. The script's `--output-dir` default is `outputs/design_exercise`, matching the paper. 2. The paper prose on line 56 of `06_design_comparison.tex` says squared-distance gives the narrowest maximum-coordinate interval "for every alpha <= 3". A minor mismatch: the paper rounds the whole-grid mean coordinate widths to "0.102, 0.104, 0.102" for Discrete, Squared, Manhattan; the data give 0.1025, 0.1042, 0.1021 — Discrete-metric rounds to 0.103, not 0.102, on standard half-up rounding. Either re-round or report at 4 decimals.

`CLAUDE.md` still describes a four-rule headline; this is explicitly flagged as stale in `_context/current_status.md`. The code, paper, and outputs are all consistent with the three-rule (discrete, squared, Manhattan) design dictated by ADR-0001's fourth amendment. The four-rule rubric in the reviewer mandate therefore does not match the project's current scope; the audit applies it to the three rules that are actually run.

## 3. Correctness concerns

- `design_efficiency.py:RULES = ("discrete", "quadratic", "manhattan")` — Hamming is intentionally excluded. The decision is justified by `outputs/verification/hamming_spike.md` (full-run projection ~341 hours, sandwich/sample containment FAIL).
- Manhattan coordinate bounds use `manhattan_sharp_coordinate_bounds` (a `brentq` root-find over the threshold c); the grid-based coordinate bounds returned by `manhattan_threshold_bounds` are computed but immediately overridden (`_grid_coord, linear, skewed, feasible = manhattan_threshold_bounds(...)` then `coord = manhattan_sharp_coordinate_bounds(r, n, k)`). This is correct but wasteful: the grid coordinate computation is dead work whenever the sharp version is called. Consider extracting the linear/mean grid loop into its own function so the coordinate grid columns are not built.
- `_MANHATTAN_INV_CACHE` is a module-level mutable dict keyed by `(n, len(c_grid))`. Safe within a single run; a second `BoundComputer` instantiated with a different tolerance but the same grid length would silently reuse the previous cache. Not a concern for the CLI but worth a comment.
- `split_winner_credit(values, tol=1e-12)` uses an absolute tolerance against the minimum. With widths around 0.10–0.18 this is essentially exact tie detection. Recorded tie rate is 0.0000 across all cells, so the policy is moot in practice — but it would break (rounding off) if interval widths grew or coarser comparisons were ever introduced.
- `manhattan_report` builds an `O(k n)` increment list and full sort. For n=50, k=10 that is 500 items, trivial — fine. The cache in `BoundComputer.bounds` keys on `(rule, n, k, r)` so repeated reports across draws are amortized.
- `verify_regions.py` is deterministic enumeration / grids — no randomness, all 13 checks pass.

Four-rule code-review checklist applied to the three implemented rules:

- The headline comparison in `RULES` and in the paper covers quadratic / discrete / Manhattan. Hamming is documented as excluded. Chebyshev is not in code.
- The metrics in the horse race are `coord_avg`, `coord_max`, `mean_linear`, `mean_skewed` — symmetric across the three rules.
- `payment_probability` is set to `math.nan` for non-discrete rules and is explicitly excluded from `update_aggregate`'s payment list except for discrete. The paper's Payment-Probability paragraph refers only to the discrete-metric rule. This complies with the project guideline that exact-correct-report payment is not a horse-race metric.

## 6. Mathematical claims covered by code

- Discrete-metric direct optima = multinomial modes (`verify_regions.py`).
- Discrete transfer region = mode region on grid.
- Discrete coordinate interval formula contains tested grid; endpoints satisfy region.
- Squared-distance direct optima = projection of np; halfspace/transfer = projection on grid; closed-form coordinate intervals = LP intervals (`squared_closed_form_bounds` vs `squared_lp_bounds`).
- L1 direct optima = separable binomial objective; CDF increment formula; exchange region = direct optima on grid; binary median intervals.
- Direct-monetary risk-aversion counterexamples.
- Hamming binary = exact match (binary case is the only Hamming statement currently in the paper's appendix).
- Corollary 1: average coordinate widths W_0 (discrete = constant) and W_Q(m,n,k) (squared, parameterised by report concentration m), and the inequality W_Q(1) < W_0 < W_Q(k), checked over (n,k) in {(2,5,10,20)} × {2,3,5,10} (`consistency_check.py`).

## 7. Mathematical claims not covered by code

- Lemma (single-transfer characterization, `app:lemma-proof`): proof is paper-only. No code asserts it for all rules, only its instantiations.
- Manhattan single-transfer sufficiency: shown in the appendix for Manhattan because each coordinate loss is discrete-convex. `verify_regions.py` checks the exchange region against direct optima on a small grid; this is finite consistency, not proof.
- Manhattan sharp coordinate bound theorem (the threshold root-find): the formulas `phi_sup`/`psi_inf` in `manhattan_sharp_coordinate_bounds` rely on the monotonicity claim "phi_sup(c) is decreasing in c" stated in the docstring and asserted in `_context/exploration/bounds_search_manhattan_hamming.md`. No code unit test exercises this monotonicity directly. Suggested unit: random `(r,n,k)`, sample c-values in [0,1], assert phi_sup, psi_inf are monotone decreasing.
- Quadratic mean-bound LP `quadratic_transfer_mean_bounds`: uses the same single-count transfer halfspace polytope. No separate paper-side closed-form claim for the mean, so the LP is the definition. No regression test pins specific values.
- The "interpretation" claims in the paper (e.g., "Manhattan is the regime-robust choice") are interpretive statements over the same data; reproducible from `rule_comparison.csv`.

4. Add a regression test that re-runs `--smoke --validate` and asserts a small fixed CSV; the existing `--validate` flag exercises the report generators but does not pin outputs.
5. Add a unit test for `manhattan_sharp_coordinate_bounds` monotonicity (phi_sup decreasing in c, psi_inf decreasing in c) and verify against `manhattan_threshold_bounds` grid results on small (n,k).
6. Mention in `verify_regions.py`'s output that the Hamming check is binary-only and that interior Hamming non-sufficiency is documented in `outputs/verification/hamming_interior_search.md`.

Paper text:

8. `06_design_comparison.tex` line 56: change "for every alpha <= 3" to "for every alpha <= 1" (or "for alpha small enough"); 0.442 at alpha=3 is not a majority.
